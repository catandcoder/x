golang之string类型变量操作的原子性

Jul 19, 2018点击

在并发场景下，string跟interface一样，都是需要使用atomic包来保证读写的原子性。



按预期, ch 不可能被写入,因为 b 的值只可能是 “0” 或 “aa”.

但实际运行:

(即两个协程,一个不停对a重新赋值,另一个选出a的值不为"0"或"aa"的,存入一个channel,并打印. 当有10次不为"0"或"aa"后,结束打印,程序运行完成)


b的值为什么会存在 "05"或者"a"的情况 ??


这是因为,string类型并不是并发安全的.


string的内部结构：

对 string 赋值,并不是原子操作,而是会分为两步,(其实是对上面这个struct ，Go 无法保证原子性地完成赋值)

因此可能会出现 goroutine 1 刚修改完指针（str）、还没来得及修改长度（len），goroutine 2 就读取了这个string 的情况。

str 改了、len 还没来得及改,即 str="aa",len=1, 和str="0",len=2, 对应"莫名其妙"出现的 “a"和"05”

思考:

str="aa",len=1,变为"a"容易理解,但str="0",len=2为什么会是"05"呢?
